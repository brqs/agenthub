## 1. 产品概述

### 1.1 一句话定义（v1.1）

> **AgentHub —— 像聊微信一样，让多个真 Agent（外部 Claude Code / Codex + 团队自建）协作完成代码、网页、文档等可交付产物。**

### 1.2 详细描述

AgentHub 是一个以 **IM 聊天为核心交互范式** 的多 Agent 协作平台。用户像使用飞书 / 微信一样，通过新建对话、发送消息的方式与不同 AI Agent 交互。

**v1.1 重要区分**：这里的 "Agent" 是**具备工具与执行能力的真 Agent**（能写文件、跑命令、用工具），而非传统聊天机器人：
- **外部 Agent**：嵌入 Claude Agent SDK / OpenAI Agents SDK / OpenCode CLI，复用其原生 agent loop、工具或 CLI runtime
- **自建 Agent**：团队实现完整 framework（AgentLoop + ToolRegistry + MCP + ContextManager + MemoryManager），用户可对话式创建（设定 System Prompt + 工具白名单）

每个 Agent 是一个"聊天对象"，用户可以单聊、群聊、并行管理多个会话。复杂任务由主 Agent（Orchestrator）自动拆解并分派给最合适的子 Agent。所有产物（代码、网页、Diff、文档）实时写入会话级 Workspace 沙箱，**前端可在聊天流中直接预览、二次编辑、续聊**。

### 1.3 产品定位三要素（v1.1）

| 要素 | 内容 |
|------|------|
| **目标用户** | 开发者、产品经理、内容创作者等需要频繁与 AI 协作完成任务的知识工作者 |
| **核心场景** | 多任务并行 + 多 Agent 协同（写代码 + 写文案 + 设计 UI 一站搞定）+ 产物可二次编辑 |
| **核心价值** | "0 学习成本" 的多 Agent 协作体验 + **真 Agent 能力**（写文件/跑命令/用工具）+ 产物即时可视且可编辑 + 上下文连续不断 |

### 1.4 设计哲学

```
熟悉感（IM 范式）  ×  专业感（开发者工具）  ×  协作感（多 Agent 编排）
       ↓                    ↓                       ↓
   零学习成本          产物可视化              复杂任务自动化
```

## 2. 市场与竞品分析

### 2.1 现有方案的痛点

| 现有方案 | 痛点 |
|---------|------|
| **ChatGPT / Claude 网页版** | 只能与单个 Agent 对话；无群聊；产物预览弱；无法切换上下文（一开新对话就丢历史） |
| **Cursor / Claude Code（IDE 内）** | 必须在编辑器里；无 IM 体验；非开发者用不来 |
| **Coze / Dify（工作流平台）** | 配置式工作流复杂；不适合临时性、对话式任务 |
| **Slack + Bot 集成** | Bot 是被动响应；无多 Agent 协调；产物展示弱 |

### 2.2 我们的差异化

| 维度 | ChatGPT | Cursor | Coze | **AgentHub** |
|------|---------|--------|------|--------------|
| IM 体验 | ⭐ | ❌ | ❌ | ⭐⭐⭐ |
| 多 Agent 群聊 | ❌ | ❌ | ⭐⭐（需配置） | ⭐⭐⭐ |
| 产物内联预览 | ⭐⭐ | ⭐⭐⭐（仅代码） | ⭐ | ⭐⭐⭐ |
| 多会话并行 | ⭐⭐ | ❌ | ⭐⭐ | ⭐⭐⭐ |
| 学习门槛 | 低 | 中 | 高 | **极低** |
| 接入主流 Agent | ❌ | ❌ | ⭐⭐ | ⭐⭐⭐ |

### 2.3 灵感来源

- **飞书 / 微信**：IM 交互范式
- **Discord**：多频道并行讨论
- **Slack + Claude / GPT Bot**：群聊中 @ AI
- **Claude Artifacts**：产物预览
- **Notion AI**：内联富媒体

## 3. 用户画像与场景

### 3.1 主要用户画像

#### 画像 A：独立开发者「小李」
```
年龄：27 岁
职业：全栈开发者，独立做副业项目
日常工具：VS Code、ChatGPT、Cursor、Figma
痛点：
  - 一个副业项目要同时用多个 AI：写代码用 Claude，做 PPT 用 GPT，画图用 MJ
  - 每次切窗口、复制上下文、贴代码，效率低
  - 想要一个"统一的工作台"
对 AgentHub 的期待：
  - 在一个会话里同时调用多个 AI
  - 产物自动整合
  - 历史可追溯
```

#### 画像 B：产品经理「Anna」
```
年龄：30 岁
职业：互联网公司 PM
日常工具：飞书文档、Figma、ChatGPT
痛点：
  - 写需求文档要找 GPT、画原型要找设计同学
  - 自己写代码 demo 不会，但又想快速验证想法
  - AI 工具太多，每个都要重新学
对 AgentHub 的期待：
  - 用熟悉的 IM 范式，不用学
  - 让多个 AI 帮我从想法到原型一站完成
  - 产物能直接看效果
```

#### 画像 C：技术博主「Mike」
```
年龄：25 岁
职业：技术内容创作者
日常工具：Notion、Markdown、ChatGPT
痛点：
  - 写技术文章需要"代码 + 解释 + 配图 + 排版"
  - 一个 AI 干不了所有事
  - 总是在多个工具间切来切去
对 AgentHub 的期待：
  - 让代码 AI、写作 AI、设计 AI 协作产出一篇完整的文章
```

### 3.2 核心使用场景

#### 场景 1：独立开发 Todo App（小李）
```
小李打开 AgentHub →
新建群聊 "Todo App 项目"，邀请 Claude Code + Codex Helper + OpenCode Helper →
发消息："@Orchestrator 我想做一个 React + Express 的 Todo App，
       帮我设计架构、写后端、写前端、给个 logo 建议" →

Orchestrator 拆解任务：
  - @Claude Code 设计架构 → 输出架构图 + 接口说明
  - @Codex 写后端 → 输出 Express 代码（内联高亮）
  - @Claude Code 写前端 → 输出 React 组件（内联预览）
  - @OpenCode Helper 做独立验证和补充修改 → 输出修复建议

小李在聊天流里直接看到所有产物，复制代码，开始干活。
```

#### 场景 2：写技术文章（Mike）
```
Mike 打开 AgentHub →
单聊自建 "Article Agent"，发："我想写一篇 React 性能优化的文章" →
Article Agent 输出大纲 →
Mike：" 第三节展开，要有代码示例" →
Article Agent 输出第三节（含 React 代码块） →
Mike：" @Claude Code 帮我把这段代码改成 React.memo 版本" →
Claude Code 输出修改后的代码（带 Diff 视图） →
Mike 一键复制到博客。
```

#### 场景 3：临时验证想法（Anna）
```
Anna 在产品会议上突然有个想法 →
打开手机上的 AgentHub PWA →
新建会话："@Claude Code 帮我快速写一个 landing page mockup，
        包含 hero、features、CTA 三个部分" →
Claude Code 流式输出 HTML 代码 + 内联网页预览卡片 →
Anna 在 iframe 里直接看效果 →
"颜色换成蓝色调" →
Agent 输出新版本 + Diff 视图 →
满意，截图发到群里。
```

## 4. 产品价值主张

### 4.1 用户价值（v1.1）

| 价值点 | 说明 |
|--------|------|
| **零学习成本** | IM 范式人人会用，5 秒上手 |
| **多任务并行** | 多会话像多个聊天窗口，互不干扰 |
| **多 Agent 协作** | 群聊 @Orchestrator 自动调度，告别"复制上下文" |
| **真 Agent 能力** ✨ | Agent 自己写文件、跑命令、用 MCP 工具，不止聊天 |
| **产物即时可视且可编辑** ✨ | 代码 / 网页 / Diff 直接在聊天流预览；选中代码可 Monaco 编辑后续聊 |
| **上下文连续 + 长期记忆** | 历史完整保留；关键消息 pin 为长期记忆 |
| **生态开放** | 接入 Claude Code / Codex 等市面 agent runtime；用户可自建 Agent（System Prompt + 工具白名单 + MCP server） |

### 4.2 业务价值（远期）

- 通过 Orchestrator 降低用户调度多 Agent 的成本，提升用户粘性
- 统一适配器层（External / Builtin / ModelGateway）降低接入新 runtime / 新 LLM 的边际成本
- 用户自建 Agent 形成"长尾内容"
- Workspace 沙箱沉淀用户产物，提高迁移成本

### 4.3 差异化矩阵（v1.1 新增）

| 维度 | ChatGPT | Cursor / Windsurf | Devin | **AgentHub** |
|---|---|---|---|---|
| 多 Agent 协作 | ✗ | ✗ | 单 Agent | ✅ 群聊 + Orchestrator |
| 多 Runtime 接入 | 自家 | 自家 + 外接 | 自家 | ✅ Claude Code + Codex + 自建 |
| 自建 Agent Framework | ✗ | ✗ | ✗ | ✅ Loop + Tool + MCP + Memory |
| 产物即时可视 | 部分 | IDE | ✗ | ✅ iframe + 文件树 |
| 产物二次编辑 + 续聊 | ✗ | ✅ IDE 内 | ✗ | ✅ 聊天内 Monaco |
| IM 多会话并行 | 有 | ✗（IDE） | ✗ | ✅ 微信式 |

### 4.4 价值 USP（Unique Selling Proposition）

> "**唯一一个能让你像群聊一样与多个真 Agent（自带工具与执行能力）协作的平台。**"

## 6. 核心用户旅程

### 6.1 首次使用旅程（New User Journey）

```
[访问首页] ──→ [注册] ──→ [登录] ──→ [欢迎引导] ──→ [选择第一个 Agent]
                                          │
                                          ▼
                              [预置 Demo 会话] ←─ 引导用户发送第一条
                                          │
                                          ▼
                                  [看到流式回复] ─ 第一次"惊喜时刻"
                                          │
                                          ▼
                              [代码自动高亮 + 复制按钮] ─ 第二次"惊喜"
                                          │
                                          ▼
                                    [创建第二个会话]
                                          │
                                          ▼
                                  [发现侧边栏多会话]
                                          │
                                          ▼
                                  [尝试群聊 + Orchestrator]
                                          │
                                          ▼
                                      [建立心智模型]
                                      "这是个 IM"
```

**关键设计原则**：让用户在 **2 分钟内** 经历至少 2 个"惊喜时刻"。

### 6.2 单聊核心旅程

```
1. 用户进入主页（已登录）
2. 点击"+ 新建会话"
3. 弹窗：选择 Agent（默认推荐 Claude Code）
4. 输入标题（或留空自动用 Agent 名字）
5. 进入聊天窗口（侧边栏自动选中该会话）
6. 在输入框输入消息 → Enter 发送
7. 看到自己的消息上屏
8. 看到 Agent "正在输入..." 提示
9. 看到 Agent 回复逐字流式出现
10. 出现代码块 → 自动高亮 → 显示复制按钮
11. 用户点击复制 → 显示"已复制"提示
12. 用户继续追问 / 修改 / 重新生成
```

### 6.3 群聊 Orchestrator 旅程

```
1. 用户点击"+ 新建群聊"
2. 弹窗：多选 Agent（至少 2 个）
   提示："建议包含 Orchestrator 作为主协调员"
3. 自动添加 Orchestrator → 进入群聊窗口
4. 顶部显示：所有 Agent 头像叠加（如群聊）
5. 用户输入："@Orchestrator 帮我做个 Todo App"
6. Orchestrator 回复"任务规划"卡片：
   📋 拆解为 3 个子任务：
   1. @Claude Code → 设计 API
   2. @Codex → 写后端代码
   3. @Claude Code → 写前端组件
7. 看到聊天流中切换提示："→ Claude Code 接管中..."
8. Claude Code 头像出现，开始输出
9. 依次看到所有 Agent 接力工作
10. 最后看到 Orchestrator 的总结
```

### 6.4 自建 Agent 旅程

```
1. 用户点击侧边栏底部"⚙ Agent 管理"
2. 进入 Agent 列表页（内置 Agent + 自建 Agent 分组）
3. 点击"+ 创建 Agent"
4. 表单：
   - 头像（选择 / 上传）
   - 名称（如"文案专家"）
   - Runtime Provider（Claude Code / Codex / OpenCode / Builtin）
   - System Prompt（文本框）
   - 能力标签（多选）
   - Builtin 配置（model_backend / max_iterations / MCP servers）或 runtime 专属配置（高级选项）
5. 点击"测试"按钮 → 弹出试用对话框

```

## 10. Orchestrator 协作设计

> ✏️ v1.1 注：Orchestrator 框架不变，但子 Agent 升级为**真 Agent**（外部 Claude Code / Codex 或自建 BuiltinAgent）。同一会话的所有子 Agent 共享一个 Workspace 沙箱，可以协作产出文件；前端在 Orchestrator 任务卡上显示"工具调用次数 + 产物文件列表"。

### 10.1 Orchestrator 角色定位

```
Orchestrator = 群聊中的"组长"
├─ 接收用户的复杂请求
├─ 拆解为子任务（每个子任务指派给一个真 Agent）
├─ 分派给最合适的子 Agent（External / Builtin）
├─ 监督执行（call_id 重映射为 task_id.<原 call_id> 防冲突）
├─ 失败降级（任务失败不阻塞独立任务，依赖失败任务 skipped；全失败仍出 summary）
└─ 汇总产出（含 workspace 文件列表）
```

### 10.1.1 v1.1 群聊示例

```
用户：@Orchestrator 帮我做一个 React Todo App，前端 + 后端 + 一键预览

Orchestrator → 拆解为 3 个子任务：
  task-1 @claude-code     "写 React 前端到 frontend/"
  task-2 @codex           "写 FastAPI 后端到 backend/"
  task-3 @builtin-deploy  "把 frontend/ 跑起来并返回预览 URL"   (depends_on: [task-1])

Claude Code → 真实写入 workspace 5 个 .tsx 文件 → 前端实时显示 5 个 ToolCallBlock + 文件树更新
Codex      → 真实写入 workspace 3 个 .py 文件 + 跑 alembic init → ToolCallBlock + tool_result
BuiltinAgent → tool: bash("pnpm install && pnpm dev") → tool_result(ok, url=http://localhost:5173)
             → 前端在 ArtifactPreview 中 iframe 这个 URL

用户在 Monaco 改了 App.tsx → 自动续聊 "我把 App.tsx 改成了这样，请基于此继续优化" → Agent 接续
```

## 14. 指标与北极星

### 14.1 北极星指标（North Star Metric）

> **DAU 用户的每日会话数（Conversations per DAU）**

理由：直接反映"用户依赖 AgentHub 完成多任务"的核心价值。

### 14.2 关键指标

#### 活跃指标
- **DAU**：日活跃用户
- **WAU / MAU**：周 / 月活跃
- **会话数 / DAU**：核心使用深度
- **群聊会话占比**：体现产品独特价值

#### 体验指标
- **首次"惊喜时刻"耗时**：注册到看到第一次流式回复
  - 目标：< 90s
- **SSE 首 token 延迟 P95**：< 2s
- **消息发送成功率**：> 99%
- **流式中断率**：< 1%

#### 留存指标
- **次日留存** / **7 日留存** / **30 日留存**

#### 比赛专属指标
- **Demo 流畅度**：演示无卡顿、无报错
- **AI 协作记录完整度**：Spec / Skill / Rules 沉淀完整
- **代码理解度**：每人能讲清自己模块
